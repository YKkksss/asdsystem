## 问题分级

### 高：健康详情接口存在“内网来源即放行”的路由级越权与信息泄露

- 结论：不通过
- 影响：匿名内网访问或低权限已登录用户都能读取数据库、Redis、存储使用率等系统健康细节，属于管理/调试信息泄露；这会直接削弱接口最小权限原则，也使前端统一入口上的任何内网终端都能探测后端基础设施状态。
- 理由：`SystemHealthDetailView` 使用 `HasConfiguredSystemPermissionOrInternalRequest`；而该权限类在认证失败或权限不满足时，只要 `_is_internal_request(request)` 为真就返回 `True`。从宿主机或局域网访问时，客户端 IP 通常就是私网地址，因此会被直接放行。
- 证据：`fullstack/backend/apps/system/api/views.py:25`、`fullstack/backend/apps/system/api/views.py:30`、`fullstack/backend/apps/common/permissions.py:61`、`fullstack/backend/apps/common/permissions.py:71`、`fullstack/backend/apps/common/permissions.py:163`、`fullstack/backend/apps/common/permissions.py:183`、`.tmp/acceptance5_health_detail_public.json:1`、`.tmp/acceptance5_borrower_health_detail_remote.json:1`、`.tmp/acceptance5_health_detail_public_remote.json:1`、`.tmp/acceptance5_borrower_health_detail_public_remote.json:1`
- 可复现验证方式：
  - 命令 1：`curl http://127.0.0.1:8080/api/v1/system/health/detail/`
  - 预期结果：当前实现错误返回 `200`，并包含 `database / redis / storage` 详情。
  - 命令 2：`curl -H 'X-Forwarded-For: 8.8.8.8' http://127.0.0.1:8080/api/v1/system/health/detail/`
  - 预期结果：返回 `401`，说明权限实际上被“内网来源”条件绕过。
- 最小可执行改进建议：将“内网来源豁免”从通用权限基类中移除，`/health/detail/` 改为仅管理员或明确授权的运维权限可访问；若确需内网探针能力，应改为单独的只读探针接口并基于可信反向代理或固定源地址白名单收口，而不是对所有私网来源放行。

### 高：超期催还邮件异步发送存在事务竞态，真实部署下邮件会静默卡在 `PENDING`

- 结论：不通过
- 影响：站内通知虽然成功，但邮件提醒在真实异步部署下可能永远不发送，直接影响 Prompt 中“站内通知 + 邮件”的业务要求，也会造成运维误判，以为提醒已创建但实际上未送达。
- 理由：`dispatch_borrow_reminder_for_application()` 在事务中调用 `create_email_reminder_record()`；该函数创建 `EmailTask` 后立即执行 `dispatch_email_task(email_task)`，而 `dispatch_email_task()` 在非 eager 模式下直接 `delay()`。如果 worker 抢在事务提交前消费任务，`process_email_task()` 读取不到刚创建的 `EmailTask`，就会直接 `return`，导致 DB 里的任务和提醒记录长期停留在 `PENDING`。
- 证据：`fullstack/backend/apps/borrowing/services.py:519`、`fullstack/backend/apps/borrowing/services.py:530`、`fullstack/backend/apps/borrowing/services.py:549`、`fullstack/backend/apps/borrowing/services.py:556`、`fullstack/backend/apps/notifications/services.py:183`、`fullstack/backend/apps/notifications/services.py:186`、`fullstack/backend/apps/notifications/services.py:232`、`fullstack/backend/apps/notifications/services.py:240`、`.tmp/acceptance5_runtime_borrow_reminder_dispatch.json:1`、`.tmp/acceptance5_runtime_borrow_reminder_state.json:11`、`.tmp/acceptance5_runtime_borrow_reminder_state.json:36`、`.tmp/acceptance5_runtime_borrow_reminder_state_after_wait.json:11`、`.tmp/acceptance5_runtime_borrow_reminder_state_after_manual_dispatch.json:11`、`.tmp/acceptance5_runtime_borrow_reminder_state_after_manual_dispatch.json:36`
- 可复现验证方式：
  - 步骤 1：创建一条已超期借阅并触发 `POST /api/v1/borrowing/reminders/dispatch/`
  - 步骤 2：检查 `BorrowReminderRecord` 与 `EmailTask` 状态
  - 预期结果：当前实现会看到站内提醒 `SUCCESS`，邮件提醒和 `EmailTask` 都停留 `PENDING`，等待后仍不变化；手工再次执行一次 Celery 任务后才转为 `SUCCESS`
- 最小可执行改进建议：把 `send_email_task.delay(email_task.id)` 移到 `transaction.on_commit()` 中；同时在 `process_email_task()` 读取不到记录时应 `retry` 或记录失败，而不是静默 `return`；`BorrowReminderRecord.sent_at` 也应只在邮件真正成功后写入。

### 中：自动化测试未覆盖两条高风险真实场景，出现“全绿但仍带高风险缺陷”
