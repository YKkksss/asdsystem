# 对原始 Prompt 的疑问问题记录

1. 审批规则问题
   - 问题：项目角色只有“管理员 / 档案员 / 借阅人 / 审计员”，但借阅流程中又出现了“领导审批”，未明确“领导”在系统中的身份定义，以及是一层审批还是多层审批。
   - 我的理解：领导不是独立角色，而是部门内具备审批权限的负责人；借阅流程先按单层审批实现。
   - 解决方式：在组织架构中为部门配置审批负责人，并增加借阅审批权限点，借阅申请提交后自动流转给所属部门负责人审批。

2. 密级与脱敏规则问题
   - 问题：项目要求“按密级自动隐藏敏感字段”，但没有明确密级枚举值，也没有明确哪些字段属于敏感字段。
   - 我的理解：至少需要统一密级等级，并对摘要、责任者、实体位置、预览入口、下载权限等字段做差异化可见控制。
   - 解决方式：定义固定密级字典和字段可见性矩阵，由后端根据角色、组织和密级返回已脱敏结果。

3. 全文检索范围问题
   - 问题：项目要求“全文检索支持按关键词 + 字段组合筛选”，但同时只明确了 PDF/图片上传和缩略图生成，没有说明扫描图片是否需要 OCR 后再纳入全文检索。
   - 我的理解：当前描述没有明确 OCR，因此全文检索应先覆盖编目字段和可直接提取文本的电子原生 PDF，图片扫描件主要依赖编目字段检索。
   - 解决方式：先实现“元数据检索 + 电子原生 PDF 文本检索”的组合方案，并在文档中明确图片扫描件不做文字识别。

4. 档号与条码二维码编码规则问题
   - 问题：项目要求支持条码/二维码生成与打印，但没有说明档号生成规则，也没有说明条码和二维码中承载的内容是否一致。
   - 我的理解：档号应作为档案唯一标识，条码和二维码都基于档号生成，打印标签展示档号和题名等必要信息。
   - 解决方式：统一定义档号规则、条码二维码编码内容和打印模板，避免后续入库、检索和定位规则不一致。

5. 借阅单粒度问题
   - 问题：项目描述提到“线上申请、领导审批、出入库登记”，但没有明确一张借阅申请单是对应一份档案，还是允许一次申请多份档案。
   - 我的理解：为了保证审批、出入库、归还、超期统计简单清晰，先按“一张借阅单对应一份档案”处理更稳妥。
   - 解决方式：借阅模块按单档案申请设计，若用户需要批量借阅，则由前端批量创建多条借阅申请。

6. 借阅期限与催还策略问题
   - 问题：项目明确要求“到期归还、超期自动催还（站内通知 + 邮件）”，但没有说明借阅期限是固定天数还是申请人填写，也没有说明提醒频率。
   - 我的理解：预计归还日期应由申请时填写，系统基于预计归还日期执行提醒；催还至少要覆盖到期前提醒、到期提醒和超期提醒。
   - 解决方式：在借阅申请中强制填写预计归还日期，并设定统一的催还策略，例如到期前提醒一次、到期当天提醒一次、超期后按天提醒。

7. 归还凭证校验问题
   - 问题：项目描述为“归还需拍照或上传交接单”，但未说明是二选一即可，还是两者都要；也未说明谁来确认归还完成。
   - 我的理解：照片和交接单二选一即可，但必须至少上传一种；归还完成应以档案员确认入库为准。
   - 解决方式：归还接口强制校验至少上传一类凭证，提交后由档案员执行归还确认，确认完成后再恢复档案在库状态。

8. 销毁审批边界问题
   - 问题：项目提到“销毁留痕”，但没有明确销毁由谁发起、谁审批、执行前要满足什么前置条件。
   - 我的理解：销毁应由档案员发起，管理员审批，且借出中的档案不得销毁。
   - 解决方式：建立销毁申请、审批、执行三段流程，并要求填写销毁原因、依据、审批意见和执行记录，销毁记录永久保留。

9. 审计日志范围与不可修改实现问题
   - 问题：项目要求“所有关键操作写入不可修改审计日志”，但没有明确哪些操作属于关键操作，也没有说明“不可修改”是业务层不可改还是数据库层不可改。
   - 我的理解：登录、权限变更、档案增改、上传预览下载、借阅审批、出入库、归还、销毁、报表导出都属于关键操作；不可修改应至少覆盖应用层和数据库层。
   - 解决方式：明确关键操作清单，审计日志表只允许新增，不提供修改删除接口，并在数据库权限层禁用更新和删除。

10. 组织数据权限范围问题
   - 问题：项目支持组织架构与角色权限，但未明确不同角色跨部门查看数据的范围，例如借阅人是否能检索全库、审计员是否可查看全部组织日志。
   - 我的理解：借阅人默认只查看其有权限访问的档案和本人申请记录；审计员可以查看审计日志和报表，但不修改数据；管理员拥有全局数据权限。
   - 解决方式：按角色定义数据范围矩阵，并在后端统一执行数据过滤，避免只在前端做页面可见性控制。

11. 中文界面范围问题
   - 问题：项目只写了“界面中文”，但没有明确中文要求覆盖哪些范围，例如菜单、表单提示、校验消息、通知内容、邮件内容是否全部中文。
   - 我的理解：既然系统面向机关与企业档案管理场景，前端界面和系统提示应统一使用中文，通知和邮件模板也应默认中文。
   - 解决方式：在前端界面、接口错误提示、站内通知模板和邮件模板中统一采用中文文案，并在设计阶段同步整理文案清单。

12. 扫描上传文件大小阈值问题
   - 问题：项目要求“上传前校验文件扩展名和文件大小”，但没有明确单文件允许的最大大小，也没有区分 PDF 与图片是否采用不同阈值。
   - 我的理解：为了先落稳扫描链路，当前可以采用统一阈值控制，避免前后端出现不同校验口径。
   - 解决方式：当前实现先采用“单文件默认不超过 50 MB”的统一规则，并通过配置项保留后续调整空间；若业务侧后续给出更明确要求，再按文件类型细化限制。
