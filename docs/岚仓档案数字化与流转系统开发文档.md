# 岚仓档案数字化与流转系统开发文档

## 1. 文档定位

本文档用于指导“岚仓档案数字化与流转系统”从 0 到 1 的需求分析、系统设计、技术选型、模块拆解、数据建模、接口规划与实施落地。文档目标不是只做概念说明，而是直接作为后续立项、排期、原型设计、数据库设计、接口开发和测试验收的统一基线。

## 2. 项目概述

### 2.1 项目名称

岚仓档案数字化与流转系统

### 2.2 项目目标

为机关与企业提供覆盖纸质档案全生命周期的一体化平台，实现以下目标：

1. 支持纸质档案从入库、扫描、编目、上架、检索、借阅、归还到销毁的闭环管理。
2. 提升档案定位效率、借阅流转效率和审计追溯能力。
3. 通过权限控制、密级脱敏、水印预览、下载留痕等方式增强档案安全。
4. 通过报表分析掌握档案利用情况、借阅风险和管理质量。

### 2.3 核心业务特征

1. 同时管理纸质档案与数字化副本。
2. 需要兼顾“实体档案位置管理”和“电子文件在线访问”。
3. 需要严格的角色权限和审计留痕。
4. 借阅流程涉及申请、审批、出入库登记、超期提醒、归还确认。
5. 检索结果要受到密级和字段可见范围约束。

## 3. 建设范围与边界

### 3.1 建设范围

1. 组织架构与角色权限管理。
2. 账号登录、失败锁定、解锁恢复。
3. 档案入库、条码/二维码生成打印、柜架位置管理。
4. 扫描任务分配、批量上传 PDF/图片、缩略图生成。
5. 编目字段校验与档案状态流转。
6. 全文检索与字段组合筛选。
7. 检索结果按密级自动隐藏敏感字段。
8. 借阅申请、领导审批、出入库登记。
9. 超期催还、站内通知、邮件提醒。
10. 归还凭证上传（照片或交接单）。
11. 销毁申请、审批、执行留痕。
12. 借阅次数、超期率、利用率报表及导出。
13. 在线预览水印、下载二次确认、下载用途记录。
14. 不可修改审计日志。

### 3.2 设计原则

1. 优先完成业务闭环，不做过度微服务拆分。
2. 尽量减少外部依赖，当前采用单体架构更合适。
3. 只围绕当前项目描述范围进行设计，不引入与本项目无关的扩展能力。

## 4. 角色与权限设计

### 4.1 基础角色

| 角色 | 说明 | 核心权限 |
| --- | --- | --- |
| 管理员 | 系统全局管理者 | 组织、账号、角色、字典、参数、全部档案与日志查看 |
| 档案员 | 档案业务执行者 | 入库、编目、扫描任务、上架、出入库、归还确认、销毁登记 |
| 借阅人 | 普通使用者 | 检索、在线查看授权内容、发起借阅申请、查看本人借阅记录 |
| 审计员 | 审查监督角色 | 查看审计日志、借阅记录、销毁记录、报表导出，不可修改业务数据 |

### 4.2 审批权限补充说明

用户提供的角色里没有单独的“领导”角色，但业务要求存在“领导审批”。推荐设计为：

1. 领导审批不新增独立角色，而是通过权限点 `borrow_approve` 授予指定负责人。
2. 组织架构中为部门配置“审批负责人”。
3. 借阅申请提交后，根据申请人所属部门自动路由到对应负责人审批。
4. 当部门未配置负责人时，允许回退到管理员或档案主管审批。

这样可以减少角色膨胀，同时满足组织内不同层级审批。

### 4.3 权限控制粒度

权限建议拆为四层：

1. 菜单权限：是否可见某个页面。
2. 操作权限：是否允许新增、编辑、审批、导出、下载。
3. 数据权限：仅本人、仅本部门、跨部门、全局。
4. 字段权限：敏感字段是否可见、是否可下载原件。

### 4.4 密级控制建议

密级建议支持字典化配置，默认提供：

1. 公开
2. 内部
3. 秘密
4. 机密
5. 绝密

系统应根据用户角色、所属组织、审批状态和档案密级综合决定：

1. 能否检索到该档案。
2. 能否看到完整字段。
3. 能否在线预览。
4. 能否下载原始文件。

## 5. 核心业务流程

### 5.1 档案数字化主流程

1. 档案员创建入库记录。
2. 系统生成档案编号、条码或二维码。
3. 档案员录入实体位置信息（库房、柜、架、层、盒）。
4. 档案员或管理员创建扫描任务并分配给执行人。
5. 执行人批量上传 PDF 或图片。
6. 系统进行必填字段校验、文件格式校验、缩略图生成。
7. 档案员完成编目审核并上架。
8. 档案进入可检索、可借阅状态。

### 5.2 借阅流转流程

1. 借阅人检索档案并提交借阅申请。
2. 系统校验借阅资格、密级权限、档案当前状态。
3. 申请进入领导审批。
4. 审批通过后，档案员执行出库登记。
5. 借阅人领取档案，系统记录借出时间、预计归还时间、借阅用途。
6. 到期前与超期后由系统自动催还。
7. 借阅人归还档案，上传交接照片或交接单。
8. 档案员验收后完成入库登记，闭环结束。

### 5.3 销毁流程

1. 档案员发起销毁申请。
2. 系统校验保管期限、当前借阅状态、销毁依据是否完整。
3. 审批通过后登记销毁执行信息。
4. 系统保留销毁原因、执行人、审批人、时间、附件与审计日志。
5. 销毁完成后档案不可再借阅，但销毁记录永久保留。

## 6. 功能模块详细分析

### 6.1 组织架构与用户权限模块

#### 功能目标

实现组织、部门、用户、角色、权限点的统一管理，为全系统提供安全访问基础。

#### 主要功能

1. 组织树管理，支持多级部门。
2. 用户管理，支持启用、停用、重置密码、锁定状态查看。
3. 角色管理，支持角色与权限点绑定。
4. 数据范围管理，如“仅本人”“本部门”“本部门及子部门”“全部”。
5. 部门审批负责人配置。

#### 关键规则

1. 一个用户可属于一个主部门，可兼任多个角色。
2. 审批链默认取用户主部门的负责人。
3. 审计员默认不具备修改类权限。

### 6.2 登录认证与账号安全模块

#### 功能目标

保证系统登录安全和基础账号风控。

#### 主要功能

1. 用户名密码登录。
2. 登录失败次数统计。
3. 连续失败 3 次锁定 15 分钟。
4. 锁定剩余时间提示。
5. 退出登录、Token 刷新或会话续期。

#### 关键规则

1. 失败计数以“连续失败次数”为准。
2. 登录成功后失败计数立即清零。
3. 锁定状态优先用 Redis 保存，TTL 为 15 分钟。
4. MySQL 同步保留最后登录时间、最后失败时间、累计失败次数等审计字段。

#### 推荐实现

1. 前后端分离场景推荐 JWT 或基于 HttpOnly Cookie 的会话认证。
2. Redis 用于登录失败计数、短期锁定状态、验证码或限流扩展。
3. 管理员可手工解锁账号，操作必须写入审计日志。

### 6.3 档案入库模块

#### 功能目标

实现纸质档案基础信息录入、条码/二维码标识生成和实体位置管理。

#### 主要功能

1. 新建档案入库单。
2. 自动生成档号或按规则生成档号。
3. 生成条码、二维码并打印。
4. 记录实体位置信息。
5. 记录档案状态、保管期限、密级、责任者、形成时间等。

#### 必填字段

1. 档号
2. 题名
3. 年度
4. 保管期限
5. 密级

#### 扩展字段建议

1. 责任部门
2. 责任人
3. 关键词
4. 摘要
5. 页数
6. 载体类型
7. 备注

#### 实体位置模型

建议使用标准化位置编码：

`库房 -> 区 -> 柜 -> 架 -> 层 -> 盒`

这样便于后续快速定位、统计库容和实体移位。

### 6.4 扫描任务与数字化加工模块

#### 功能目标

支持从纸质档案到电子文件的数字化过程管理。

#### 主要功能

1. 创建扫描任务。
2. 批量分配执行人。
3. 批量上传 PDF/图片。
4. 自动生成缩略图。
5. 记录任务进度、处理结果、异常原因。
6. 将扫描文件绑定到档案记录。

#### 关键规则

1. 支持单个档案多文件上传。
2. 支持批量上传后自动归档到任务批次。
3. PDF 缩略图取首页，图片缩略图按固定尺寸生成。
4. 上传前校验文件扩展名和文件大小。

#### 实现建议

1. 支持 `pdf/jpg/jpeg/png/tif/tiff`。
2. 使用 Celery + Redis 异步生成缩略图，避免阻塞上传接口。
3. 对电子原生 PDF 可直接提取文本进入全文索引。

### 6.5 编目与档案状态管理模块

#### 功能目标

让档案信息达到可检索、可借阅、可审计的标准化状态。

#### 主要功能

1. 编目信息编辑与校验。
2. 多版本修订留痕。
3. 上架、下架、冻结、销毁状态管理。
4. 档案与实体位置、数字文件、借阅记录的关联展示。

#### 推荐状态

| 状态 | 说明 |
| --- | --- |
| 草稿 | 已创建，信息不完整 |
| 待扫描 | 已入库，待数字化 |
| 待编目 | 文件已上传，待校验 |
| 已上架 | 可检索、可借阅 |
| 借出中 | 实体档案已借出 |
| 销毁审批中 | 已提交销毁流程 |
| 已销毁 | 不可借阅，仅保留记录 |
| 冻结 | 因审计或异常临时冻结 |

### 6.6 全文检索与组合筛选模块

#### 功能目标

让用户能够快速找到目标档案，同时保证敏感信息按密级自动脱敏。

#### 检索能力

1. 关键词搜索。
2. 字段筛选：档号、题名、年度、保管期限、密级、部门、状态、位置。
3. 组合筛选：关键词 + 多字段联合查询。
4. 排序：相关度、创建时间、年度、借阅次数。

#### 检索实现建议

当前建议优先采用以下方案：

1. MySQL 普通字段索引支持精确筛选。
2. 对题名、关键词、摘要、提取文本建立 FULLTEXT 索引。
3. 通过 DRF + django-filter 统一处理组合查询。

#### 密级脱敏策略

根据用户权限对以下字段进行隐藏或部分掩码：

1. 摘要
2. 责任者
3. 文件预览入口
4. 下载按钮
5. 实体精确位置

脱敏实现应在后端完成，不能只依赖前端隐藏。

### 6.7 借阅申请与审批模块

#### 功能目标

规范借阅流程，做到“谁申请、谁审批、谁出库、谁归还”全链路可追溯。

#### 主要功能

1. 借阅申请单创建。
2. 填写借阅用途、借阅期限、附件说明。
3. 自动流转到对应领导审批。
4. 审批通过、驳回、撤回。
5. 档案员办理出库登记。

#### 核心字段建议

1. 申请编号
2. 申请人
3. 所属部门
4. 档案 ID / 档号
5. 借阅用途
6. 预计归还日期
7. 审批意见
8. 实际借出时间
9. 实际归还时间

#### 校验规则

1. 已借出、已销毁、冻结中的档案不可再次借阅。
2. 权限不足或密级不匹配时不可提交申请。
3. 同一用户存在严重超期未归还记录时，可限制再次申请。

### 6.8 出入库登记与归还管理模块

#### 功能目标

形成实体档案借出与归还的闭环台账。

#### 主要功能

1. 审批通过后执行出库登记。
2. 记录领取人、办理人、时间、位置变化。
3. 归还时上传照片或交接单。
4. 档案员确认归还质量并重新入库。
5. 系统自动恢复档案状态为“已上架”。

#### 关键规则

1. 借阅完成前档案状态为“借出中”。
2. 归还凭证至少上传一种：照片或交接单。
3. 档案员确认归还后才算流程结束。

### 6.9 超期催还与消息通知模块

#### 功能目标

减少超期借阅风险，形成自动提醒机制。

#### 主要功能

1. 到期前提醒。
2. 到期当天提醒。
3. 超期后每日提醒。
4. 站内通知。
5. 邮件通知。

#### 推荐策略

1. 到期前 3 天提醒一次。
2. 到期当天提醒一次。
3. 超期后每日 09:00 提醒一次。
4. 超期超过阈值时抄送档案员和审批负责人。

#### 实现方式

1. Celery Beat 定时扫描即将到期和已超期记录。
2. Redis 作为 Celery Broker / Cache。
3. 站内消息写入 MySQL，邮件异步发送。

### 6.10 销毁管理模块

#### 功能目标

对达到销毁条件的档案做合规、可追溯的销毁处理。

#### 主要功能

1. 销毁申请。
2. 销毁依据登记。
3. 销毁审批。
4. 销毁执行记录。
5. 销毁附件上传。
6. 销毁后锁定借阅和编辑入口。

#### 核心规则

1. 借出中的档案不允许进入销毁流程。
2. 必须填写销毁原因、依据、审批意见。
3. 销毁记录永久保留，不允许物理删除。

### 6.11 在线预览与下载控制模块

#### 功能目标

既支持业务使用，又控制电子档案外泄风险。

#### 在线预览要求

1. PDF 在线预览。
2. 图片在线预览。
3. 预览页显示动态水印：姓名 + 时间。
4. 预览权限受密级和角色控制。

#### 下载控制要求

1. 点击下载先弹出二次确认。
2. 必须填写下载用途。
3. 后端记录下载人、时间、IP、用途、文件标识。
4. 下载权限独立于预览权限控制。

#### 技术建议

1. PDF 预览可采用前端 PDF.js。
2. 图片预览可通过受控下载 URL 或后端流式输出。
3. 水印应由前端展示层和后端日志双重控制。

### 6.12 报表与导出模块

#### 功能目标

通过经营性和管理性指标评估档案利用情况与借阅风险。

#### 核心报表

1. 借阅次数统计
2. 超期率统计
3. 利用率统计

#### 指标口径建议

1. 借阅次数：统计周期内借阅单完成出库的次数。
2. 超期率：统计周期内超期借阅单数 / 到期应归还借阅单数。
3. 利用率：统计周期内被借阅过的档案数 / 可借阅档案总数。

#### 导出建议

1. 支持 Excel 导出。
2. 支持按组织、年度、密级、时间范围筛选导出。
3. 导出动作写入审计日志。

### 6.13 审计日志模块

#### 功能目标

确保所有关键操作都能被追溯，并尽可能做到不可篡改。

#### 必须记录的关键事件

1. 登录成功、失败、锁定、解锁。
2. 用户、角色、权限变更。
3. 档案新增、编辑、删除逻辑动作、状态变更。
4. 文件上传、预览、下载。
5. 借阅申请、审批、出库、归还、催还。
6. 销毁申请、审批、执行。
7. 报表导出。

#### 不可修改实现建议

1. 审计日志表只允许 INSERT，不提供更新和删除接口。
2. 应用层禁止修改审计日志。
3. 数据库账号权限层面对审计日志表禁用 UPDATE/DELETE。

## 7. 非功能需求

### 7.1 安全要求

1. 所有接口必须鉴权。
2. 密级脱敏必须后端控制。
3. 上传文件必须限制类型和大小。
4. 下载必须二次确认并记录用途。
5. 审计日志不可修改。
6. 登录失败 3 次锁定 15 分钟。

### 7.2 性能要求

1. 常规检索响应建议控制在 2 秒内。
2. 大文件上传采用分批处理或异步后处理。
3. 缩略图生成与通知发送走异步队列。
4. 报表类接口应支持分页和时间范围过滤。

### 7.3 可用性要求

1. 前端页面适配桌面端与平板端。
2. 关键表单支持草稿保存或失败恢复提示。
3. 文件上传失败要有明确的重试反馈。

### 7.4 可维护性要求

1. 后端采用模块化 Django App 组织。
2. 前端按业务域拆分页面与状态管理。
3. 统一字典、枚举、状态机定义。
4. 全链路日志、错误码和异常处理规范化。

## 8. 技术架构设计

## 8.1 总体架构

推荐采用“前后端分离 + 单体业务服务 + 异步任务”的架构：

1. 前端：Vue 3 Web 自适应管理端。
2. 后端：Django + DRF 提供 REST API。
3. 数据库：MySQL 存储业务数据。
4. 缓存与队列：Redis 用于缓存、登录锁定、异步任务 Broker。
5. 异步任务：Celery 处理缩略图生成、邮件通知、催还扫描。
6. 文件存储：采用服务器私有目录存储。

### 8.3 为什么推荐单体架构

当前系统虽然业务模块较多，但边界都围绕“档案管理”展开，采用单体架构有明显优势：

1. 研发成本更低。
2. 权限、审计、状态流转更容易统一。
3. 事务一致性更容易保证。
4. 对团队规模要求更低。

不建议一开始拆分微服务，否则会增加认证、消息、链路追踪、部署复杂度。

## 14. 检索与脱敏设计细化

### 14.1 检索方案推荐

推荐采用“两层检索”：

1. 元数据检索：对档号、题名、年度、保管期限、密级等做精确或模糊筛选。
2. 文本检索：对摘要、关键词、提取文本做 FULLTEXT 检索。

### 14.2 脱敏实现建议

后端根据当前用户计算 `field_visibility`，前端只负责展示，不做权限判断。

示例：

| 字段 | 公开/内部 | 秘密 | 机密/绝密 |
| --- | --- | --- | --- |
| 题名 | 可见 | 可见 | 可见或部分掩码 |
| 摘要 | 可见 | 部分可见 | 隐藏 |
| 实体位置 | 可见 | 柜架级 | 仅档案员可见 |
| 预览 | 可见 | 申请后可见 | 默认不可见 |
| 下载 | 需权限 | 需权限 | 需权限 |

`/data/archive-files/{year}/{archive_id}/thumb/`

## 19. 风险点与应对建议

### 风险 1：全文检索效果不足

原因：图片型扫描文件本身没有文本层，如果业务资料主要以图片形式上传，全文检索将更多依赖编目字段。

建议：

1. 强化档号、题名、年度、保管期限、密级、关键词等编目字段质量。
2. 对电子原生 PDF 提取文本，用于补充全文检索能力。

### 风险 2：权限模型过粗导致泄密

原因：只做页面级权限而没有字段级和数据级限制时，容易出现敏感信息暴露。

建议：

1. 后端统一做数据级与字段级权限裁剪。
2. 高密级档案默认最小授权。

### 风险 3：审计日志被误改

原因：如果只在业务代码层限制，而数据库权限没有隔离，仍存在被修改风险。

建议：

1. 审计日志单独表、单独权限。

### 风险 4：异步任务堆积

原因：上传文件多、缩略图生成和通知发送集中时，可能拖慢系统。

建议：

1. 缩略图和邮件全部异步化。
2. 分离 Celery Worker 队列。
