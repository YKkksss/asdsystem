# 日志监控与备份恢复说明

说明：以下命令默认先进入仓库内的 `fullstack/` 目录执行。

## 本次补齐内容

当前项目已经补齐以下运维能力：

1. Django 后端日志按时间轮转，默认输出到 `fullstack/backend/logs/`
2. 后端新增详细健康检查接口：`/api/v1/system/health/detail/`
3. `fullstack/ops/` 目录提供日志轮转、服务巡检、备份、恢复脚本
4. 生产模板目录新增 `fullstack/deploy/logrotate/asd-system.conf`

## 详细健康检查接口

接口地址：

```text
GET /api/v1/system/health/detail/
```

返回信息包含以下检查项：

1. `database`：数据库连通性与响应耗时
2. `redis`：Redis 连通性与响应耗时
3. `storage`：媒体目录所在磁盘使用率与剩余容量

`status` 字段含义如下：

1. `ok`：当前检查通过
2. `warning`：当前检查接近阈值，需尽快处理
3. `error`：当前检查失败，建议立即告警

## 日志轮转

### 后端日志

后端日志由 Django 自带 `TimedRotatingFileHandler` 处理，默认策略如下：

1. 轮转周期：每天零点
2. 保留份数：14 份
3. 日志目录：`fullstack/backend/logs/`

可通过环境变量调整：

```text
BACKEND_LOG_LEVEL=INFO
BACKEND_LOG_DIR=logs
BACKEND_LOG_ROTATION_WHEN=midnight
BACKEND_LOG_ROTATION_INTERVAL=1
BACKEND_LOG_BACKUP_COUNT=14
```

### 运行期脚本日志

若你是通过 `scripts/manage_services.sh` 启动后端、Worker、Beat，这些进程的日志会写到 `fullstack/runtime/services/logs/`。可以使用以下脚本做尺寸轮转：

```bash
./ops/rotate_runtime_logs.sh
./ops/rotate_runtime_logs.sh backend
./ops/rotate_runtime_logs.sh worker
./ops/rotate_runtime_logs.sh beat
```

相关环境变量：

```text
ASD_LOG_MAX_SIZE_MB=20
ASD_LOG_BACKUP_COUNT=7
```

### 生产环境 logrotate 模板

生产环境可直接参考：

```text
fullstack/deploy/logrotate/asd-system.conf
```

安装方式示例：

```bash
sudo cp fullstack/deploy/logrotate/asd-system.conf /etc/logrotate.d/asd-system
sudo logrotate -d /etc/logrotate.d/asd-system
```

## 监控告警

### 巡检脚本

脚本位置：

```text
fullstack/ops/monitor_services.sh
```

默认会检查：

1. 后端详细健康接口
2. Redis 端口
3. Celery Worker 进程
4. Celery Beat 进程

常用命令：

```bash
./ops/monitor_services.sh
./ops/monitor_services.sh --skip-worker
./ops/monitor_services.sh --webhook https://example.com/webhook
./ops/monitor_services.sh --email ops@example.com
```

相关环境变量：

```text
ASD_MONITOR_HEALTH_URL=http://127.0.0.1:8000/api/v1/system/health/detail/
ASD_ALERT_WEBHOOK_URL=
ASD_ALERT_EMAIL=
```

说明：

1. 当健康接口状态不是 `ok` 时，脚本返回非 0
2. 若 Redis、Worker、Beat 不可用，也会返回非 0
3. 若配置了 Webhook 或邮件地址，会在失败时尝试发送告警

### 推荐定时巡检

建议通过 `cron` 每 5 分钟执行一次：

```bash
*/5 * * * * /srv/asd-system/fullstack/ops/monitor_services.sh >> /var/log/asd-system/monitor.log 2>&1
```

## 备份策略

### 备份脚本

脚本位置：

```text
fullstack/ops/backup_system.sh
```

默认备份内容包括：

1. 数据库备份
2. `media/` 文件归档
3. 当前后端 `.env` 配置副本
4. `manifest.env` 备份清单
5. `SHA256SUMS` 校验文件（如果机器支持 `sha256sum`）

常用命令：

```bash
./ops/backup_system.sh
./ops/backup_system.sh --output-dir /srv/asd-system/fullstack/runtime/backups
./ops/backup_system.sh --env-file /srv/asd-system/fullstack/deploy/env/backend.production.env
```

说明：

1. SQLite 模式直接复制数据库文件
2. MySQL 模式通过 `mysqldump` 导出 SQL
3. 默认输出目录由 `ASD_BACKUP_DIR` 控制，未配置时使用 `fullstack/runtime/backups/`

### 推荐定时备份

建议每天凌晨 3 点执行一次：

```bash
0 3 * * * /srv/asd-system/fullstack/ops/backup_system.sh --output-dir /srv/asd-system/fullstack/runtime/backups >> /var/log/asd-system/backup.log 2>&1
```

## 恢复策略

### 恢复脚本

脚本位置：

```text
fullstack/ops/restore_system.sh
```

使用示例：

```bash
./ops/restore_system.sh --backup-dir runtime/backups/asd-backup-20260314T120000Z --yes
./ops/restore_system.sh --backup-dir /srv/asd-system/fullstack/runtime/backups/asd-backup-20260314T120000Z --restore-env --yes
```

恢复前建议顺序：

1. 停止 Django、Celery Worker、Celery Beat
2. 确认目标环境 `.env` 与备份数据库类型一致
3. 执行恢复脚本
4. 启动服务并再次执行 `./ops/monitor_services.sh`

当前恢复策略具备以下保护：

1. 必须显式传入 `--yes`
2. 若数据库引擎不一致，会直接终止
3. SQLite 数据库与 `media/` 恢复前会先保留一份 `before-restore-*` 备份

## 验收建议

建议按以下顺序做验收：

1. 执行 `./ops/backup_system.sh` 生成一份备份
2. 调用 `curl http://127.0.0.1:8000/api/v1/system/health/detail/`
3. 执行 `./ops/monitor_services.sh`
4. 让某个依赖故障后再次执行 `./ops/monitor_services.sh`，确认其返回非 0
5. 在测试环境执行一次 `./ops/restore_system.sh --backup-dir ... --yes` 演练恢复链路
