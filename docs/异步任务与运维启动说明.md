# 异步任务与运维启动说明

说明：以下命令默认先进入仓库内的 `fullstack/` 目录执行。

## 适用范围

本文档用于说明当前项目在本地联调或准生产演示环境下，如何统一启动后端服务、Redis、Celery Worker、Celery Beat，并查看运行状态与日志。

当前仓库已经提供三类脚本：

1. `scripts/start.sh`：适合单进程前台调试，直接占用当前终端。
2. `scripts/manage_services.sh`：适合后台联动运行，统一管理 PID、日志和 Redis 检查。
3. `ops/`：适合巡检告警、日志轮转、备份恢复。

## 前置要求

启动前请确保以下条件满足：

1. 已完成后端依赖同步：`cd backend && uv sync`
2. 已完成前端依赖安装：`cd frontend && npm install`
3. 已复制后端环境变量：`cp backend/.env.example backend/.env`
4. 已执行数据库迁移：`cd backend && uv run manage.py migrate`
5. 如需初始化账号，执行：

```bash
cd backend
uv run manage.py bootstrap_system --username admin --password 请替换为安全密码
```

## 脚本职责划分

### `start.sh`

适合前台单服务调试：

```bash
./scripts/start.sh 1
./scripts/start.sh 2
./scripts/start.sh 3
./scripts/start.sh 4
```

对应含义：

1. 启动前端开发服务
2. 启动后端开发服务
3. 启动 Celery Worker
4. 启动 Celery Beat

### `manage_services.sh`

适合后台联动环境：

```bash
./scripts/manage_services.sh start
./scripts/manage_services.sh status
./scripts/manage_services.sh stop
```

默认会处理以下服务：

1. Redis
2. Django 后端服务
3. Celery Worker
4. Celery Beat

## 常用命令

### 启动整套后台链路

```bash
./scripts/manage_services.sh start
```

执行逻辑如下：

1. 检查 `127.0.0.1:6379` 是否已有 Redis
2. 如果没有 Redis，则尝试通过 `docker/docker-compose.yaml` 启动 Redis 容器
3. 检查 `127.0.0.1:8000` 是否已有后端服务
4. 若后端未运行，则后台启动 Django 开发服务
5. 启动 Celery Worker
6. 启动 Celery Beat

说明：

- 如果某个端口已经被外部服务占用，脚本会直接复用，不会强行覆盖。
- 如果后端是你手工通过 `scripts/start.sh 2` 启动的，`scripts/manage_services.sh stop` 不会把它停掉。

### 分服务启动

```bash
./scripts/manage_services.sh start redis
./scripts/manage_services.sh start backend
./scripts/manage_services.sh start worker
./scripts/manage_services.sh start beat
```

### 查看状态

```bash
./scripts/manage_services.sh status
```

状态会区分为：

1. `运行中（脚本管理，PID=...）`
2. `运行中（外部服务）`
3. `未运行`

### 查看日志

```bash
./scripts/manage_services.sh logs backend
./scripts/manage_services.sh logs worker
./scripts/manage_services.sh logs beat
./scripts/manage_services.sh logs redis
```

日志说明：

1. Django、Worker、Beat 日志统一写入 `runtime/services/logs/`
2. Redis 如果是脚本拉起的 Docker 容器，会通过 `docker compose logs` 查看
3. 如果 Redis 是外部已有服务，脚本无法统一读取其日志

### 停止服务

```bash
./scripts/manage_services.sh stop
```

默认停止顺序：

1. Celery Beat
2. Celery Worker
3. Django 后端
4. 脚本拉起的 Redis

说明：

- 只会停止脚本自己记录过 PID 的服务
- 外部 Redis、外部后端不会被误停

### 重启服务

```bash
./scripts/manage_services.sh restart
```

## 运行目录说明

脚本运行后会在 `fullstack/runtime/` 下生成以下结构：

```text
runtime/services/
  logs/
    backend.log
    worker.log
    beat.log
  pids/
    backend.pid
    worker.pid
    beat.pid
    redis.managed
```

字段说明：

1. `*.pid`：仅记录脚本自己启动的进程 PID
2. `redis.managed`：标记 Redis 是由脚本通过 Docker Compose 拉起

## 验证方式

建议按以下顺序验证后台链路：

### 1. 查看统一状态

```bash
./scripts/manage_services.sh status
```

### 2. 验证后端健康检查

```bash
curl http://127.0.0.1:8000/api/v1/system/health/
```

### 3. 验证 Redis 连通性

如果 Redis 为本机服务：

```bash
redis-cli -h 127.0.0.1 -p 6379 ping
```

如果 Redis 为 Docker Compose 拉起：

```bash
cd fullstack/docker
docker compose exec redis redis-cli ping
```

### 4. 验证 Celery Worker 日志

```bash
./scripts/manage_services.sh logs worker
```

正常情况下应能看到 Worker 已完成启动并开始监听任务队列。

### 5. 验证 Celery Beat 日志

```bash
./scripts/manage_services.sh logs beat
```

正常情况下应能看到定时调度器已加载周期任务。

## 演示环境建议

如果你是在演示或验收环境下运行，建议使用如下组合：

1. 前端使用 `./scripts/start.sh 1`
2. 后台链路使用 `./scripts/manage_services.sh start`
3. 联调结束后使用 `./scripts/manage_services.sh stop`

这样做的好处：

1. 前端仍然保留热更新和前台可见日志
2. 后端、Worker、Beat 不需要手工开多个终端
3. 日志和 PID 可统一回收

## 补充运维脚本

当前仓库已经补充以下运维脚本：

1. `./ops/rotate_runtime_logs.sh`
2. `./ops/monitor_services.sh`
3. `./ops/backup_system.sh`
4. `./ops/restore_system.sh`

建议与 `manage_services.sh` 配合方式如下：

1. 开发联调时通过 `./scripts/manage_services.sh start` 启动后台链路
2. 通过 `./ops/monitor_services.sh` 做巡检与告警演练
3. 通过 `./ops/rotate_runtime_logs.sh` 处理脚本运行期日志
4. 通过 `./ops/backup_system.sh` 和 `./ops/restore_system.sh` 完成备份恢复演练

详细说明见 [日志监控与备份恢复说明.md](./%E6%97%A5%E5%BF%97%E7%9B%91%E6%8E%A7%E4%B8%8E%E5%A4%87%E4%BB%BD%E6%81%A2%E5%A4%8D%E8%AF%B4%E6%98%8E.md)。
